# ClawdCursor v0.8.0 — Final Architecture

> "One tool layer. Every brain speaks the same language."

## The Problem with v0.7.5

```
Task → Preprocessor (LLM #1) → Local Parser → Classifier →
  Action Router → Decomposer (LLM #2) → per-subtask:
    Router retry → OCR Reasoner (LLM #3, 50 steps) → Vision Filler (LLM #4, 15 iter)
```

**4 LLM calls minimum.** Preprocessor parses intent. Decomposer splits subtasks. OCR Reasoner loops with text. Vision Filler loops with screenshots. Each has its own system prompt, coordinate system, action schema, and failure modes.

The AI never sees the full task + full screen together.

When a capable external model (Opus, Sonnet, GPT-4o) drives via serve/mcp, none of this runs. The internal pipeline only exists for `clawdcursor start` — and it's the source of every bug.

## The Solution: One Pipeline, Three Entry Points

```
┌─────────────────────────────────────────────────┐
│           40 TOOLS (tools/ directory)            │
│  desktop · a11y · cdp · smart · ocr · shortcuts │
└──────────┬──────────┬──────────┬────────────────┘
           │          │          │
     ┌─────┴──┐  ┌────┴───┐  ┌──┴──────────┐
     │ serve  │  │  mcp   │  │   start     │
     │ (HTTP) │  │(stdio) │  │ (agent loop)│
     └────────┘  └────────┘  └─────────────┘
         ↑           ↑             ↑
     OpenClaw    Claude Desktop  Built-in LLM
     Cursor      Any MCP client  (for basic users)
```

All three modes execute the **exact same tools**. No separate OCR Reasoner. No separate Vision Filler. No separate Computer Use adapter. Just tools.

## The New Agent Loop (`clawdcursor start`)

```typescript
async executeTask(task: string): Promise<TaskResult> {
  const tools = getAllTools();
  const toolSchemas = toOpenAiFunctions(tools);
  const messages = [
    { role: 'system', content: SYSTEM_PROMPT },
    { role: 'user', content: task }
  ];

  for (let i = 0; i < MAX_ITERATIONS; i++) {
    // 1. Call LLM with tool definitions
    const response = await callLLM(messages, toolSchemas);

    // 2. If LLM says done → verify and return
    if (response.done) {
      return verify(task, response.evidence);
    }

    // 3. If LLM calls a tool → execute it
    for (const toolCall of response.toolCalls) {
      const tool = getTool(toolCall.name);
      const result = await tool.handler(toolCall.args, ctx);
      messages.push(toolCallResult(toolCall.id, result));
    }
  }
}
```

That's it. ~30 lines of core logic. The model:
- Calls `smart_read` or `desktop_screenshot` to perceive
- Calls `smart_click`, `smart_type`, `key_press` to act
- Calls `shortcuts_list` when it wants keyboard shortcuts
- Calls `read_screen` for structured a11y data
- Decides when it's done

No preprocessor. No decomposer. No classifier. No router. The model IS the router.

## System Prompt (for `start` mode)

```
You are ClawdCursor, a desktop automation agent on {platform}.
You have {N} tools to control this computer. Use them to complete the user's task.

WORKFLOW:
1. smart_read or desktop_screenshot to see the screen
2. Decide what to do
3. Execute with smart_click, smart_type, key_press, or other tools
4. Verify by reading/screenshotting again
5. Report done when task is complete

RULES:
- Start every task by reading the screen (smart_read first, screenshot if needed)
- Use smart_click/smart_type (text-based, fast) before mouse_click (coordinate-based)
- Use shortcuts_list to find keyboard shortcuts — they're faster than clicking
- After critical actions, verify the result (read or screenshot)
- If stuck after 3 attempts, try a completely different approach
- Never repeat the same failed action
- Report done with evidence of completion
```

The smart tools (`smart_click`, `smart_type`, `smart_read`) already have multi-layer fallback built in (OCR → A11y → CDP). They're the primary interface. `desktop_screenshot` + `mouse_click` is the vision fallback — and it's just another tool, not a separate pipeline stage.

## What This Means for Each Mode

### `clawdcursor serve` (HTTP) — No change
External AI calls tools via REST. Already works perfectly (proven with OpenClaw tonight).

### `clawdcursor mcp` (stdio) — No change
External AI calls tools via MCP protocol. Same tools, different transport.

### `clawdcursor start` (agent) — Complete simplification
Internal LLM drives the same tools via function calling. One loop. The model picks tools, we execute them. Same tools that serve/mcp expose.

**The internal agent becomes just another client of the tool layer.** Same as OpenClaw, same as Claude Desktop, same as Cursor — just running locally.

## Token Economics

**v0.7.5 (Outlook email):**
- Preprocessor: ~1K tokens
- Decomposer: ~2K tokens
- OCR Reasoner: ~30K tokens (5 OCR snapshots × 6K each)
- Vision Filler: ~50K tokens (5 screenshots)
- Total: ~83K tokens, 4 models, 10+ LLM calls

**v0.8.0 (same task):**
- smart_read → smart_click "New mail" → smart_type "To" "email" → smart_type "Subject" "text" → type_text "body" → key_press "ctrl+enter"
- ~15K tokens, 1 model, 3-4 LLM calls (one per tool batch)
- Total: **~80% cheaper**

## The Principle

ClawdCursor is **hands, not a brain.** The brain is whoever calls the tools — OpenClaw, Claude Desktop, Cursor, or the built-in agent loop. All brains get the same hands. The hands are good: smart tools with multi-layer fallback, OCR, A11y, CDP, shortcuts. The brain just needs to say what to do.
