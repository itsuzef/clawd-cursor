# ClawdCursor v0.7.5 — Architecture Specification

## Philosophy
Two brains, one compilation. Keep it simple. The fundamental concept: give an AI agent eyes and hands on a real computer, efficiently and affordably.

## Architecture: Three Clear Stages

### Stage 1: Compilation Layer (Parallel, No LLM)
Run ALL perception tools simultaneously, merge into ONE structured document:

```
[PARALLEL]
  ├── OCR Engine → text elements + bounding boxes + coordinates
  ├── A11y Bridge → UI tree + control types + automation IDs + states
  └── CDP Driver → DOM elements (browser tabs only)
         │
         ▼
  MERGER → Single structured snapshot document
           Each element has: text, type, coordinates, state, a11y metadata
```

**Output format** (sent to Text LLM):
```
SCREEN SNAPSHOT (1920x1080)
Active Window: "Google Chrome" (pid 1234)

ELEMENTS:
[1] Button "Submit" at (450, 320) — enabled, a11y:Submit_Btn
[2] TextBox "Search..." at (200, 80) — focused, editable
[3] Link "Home" at (50, 15) — enabled
[4] Text "Welcome back, user" at (400, 200) — static
...
```

**Rules:**
- ALL coordinates are real screen pixels (no DPI conversion needed)
- Elements sorted top-to-bottom, left-to-right
- A11y metadata included but secondary — OCR coordinates are primary
- No personal data stored in snapshots

### Stage 2: Text LLM (The Navigator)
- Receives the compiled snapshot + task description
- Outputs structured JSON actions: `click(x,y)`, `type(text)`, `key(shortcut)`, `scroll(direction)`, `done(evidence)`, `cannot_proceed(reason)`
- Navigates "blind" using coordinates from the snapshot
- Fast, cheap model — Gemini 2.5 Flash, GPT-4o-mini, or Kimi K2
- Loops: act → re-compile snapshot → act → until done or cannot_proceed
- Max iterations: 30 (with stagnation detection at 6 identical screens)

**When Text LLM returns `cannot_proceed`:**
- Reason is passed to Vision LLM
- Text LLM does NOT retry — immediate handoff

### Stage 3: Vision LLM (The Gap Filler)
- ONLY invoked when Text LLM signals `cannot_proceed`
- Receives: screenshot image + the failed task + reason from Text LLM
- Returns: ONLY coordinates and action (click x,y / type text)
- Does NO planning, NO decomposition — just fills the gap
- More expensive model — GPT-4o, Claude Sonnet 4, or Gemini 2.5 Pro
- Max 5 iterations, then fail gracefully

## Shortcuts & Router (Pre-Stage)
Before the pipeline, a zero-LLM shortcut resolver handles:
- `open [app]` → native launch
- `press [key]` → keyboard shortcut
- `type [text]` → direct typing
- Simple navigation commands

If the shortcut resolver handles it, NO compilation or LLM calls needed.

## Pre-Processor (One LLM Call)
For compound tasks ("open X and do Y"), one cheap LLM call decomposes into structured intent:
- What app to open
- What URL to navigate to
- What remaining task to execute

## Security
- Auth token required for all endpoints
- No personal data in JSON config files
- Config stores: provider, model names, base URLs, feature flags
- API keys in env vars or encrypted store ONLY
- Safety layer classifies actions (auto/preview/confirm)
- Blocked action patterns for dangerous operations

## Config File (.clawdcursor-config.json)
```json
{
  "textModel": { "provider": "google", "model": "gemini-2.5-flash" },
  "visionModel": { "provider": "google", "model": "gemini-2.5-flash" },
  "compilation": { "ocr": true, "a11y": true, "cdp": true, "parallel": true },
  "limits": { "maxSteps": 30, "stagnation": 6, "visionMaxSteps": 5 },
  "security": { "authRequired": true, "safetyTier": "auto" }
}
```
